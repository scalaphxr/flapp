# Единое превью наложения и свои шрифты

**Дата:** 2026-07-17
**Статус:** согласовано

## Проблема

Кадр с обложкой рисуется дважды, двумя независимыми рендерами:

- статичное превью в UI — средствами CSS (`frontend/src/pages/player/PlayerPage.tsx:1946-1958`);
- итоговое видео — фильтром ffmpeg (`backend/internal/infrastructure/youtube/ffmpeg.go:267-294`).

Совпасть они могут только случайно. Фактические расхождения:

| Параметр | CSS-превью | ffmpeg (истина) |
|---|---|---|
| Позиция заголовка | `top: 68%` | `y=h*0.70` |
| Отступ подписи | `margin-top` ≈ 112px от верха тайтла | ровно `+120px` |
| Размытие фона | `blur(22px)` в пикселях экрана | `gblur=sigma=28` при 1920 |
| Затемнение фона | `brightness(0.7)` — умножение | `eq=brightness=-0.08` — сдвиг |
| Масштаб фона | `scale(1.12)` | нет, только cover-кроп |
| Тень текста | смещение `0,3`, размытие 9.6px | смещение `3,3`, без размытия |
| Начертание | `font-weight: 800/500` | отдельные файлы bold/regular |
| Длинный тайтл | обрезается многоточием | не переносится и не обрезается |

Размытие расходится сильнее всего: `blur(22px)` считается от размера превью на экране, а `sigma=28` — от 1920px. При превью шириной ~600px это эквивалент sigma≈70 — фон в CSS размыт примерно втрое сильнее, чем в видео.

Вторая задача: в наложение нельзя подставить свой шрифт — список ограничен системными шрифтами Windows (`YT_FONTS`, `PlayerPage.tsx:1222-1234`; `FontFiles`, `ffmpeg.go:188-200`).

## Решение

### Часть 1: один рендер вместо двух

Построение фильтра выносится из `RenderStill` в общую функцию, чтобы у кадра и видео был один и тот же код, а не две согласованные копии:

```go
// ffmpeg.go
func buildChain(opts RenderOpts, tmpPrefix string) (chain string, cleanup func())
```

`cleanup` удаляет временные `.title.txt`/`.sub.txt` — сейчас это делают `defer os.Remove` внутри `RenderStill`, при выносе их владельцем становится вызывающий.

Потребители:

- `RenderStill` — без изменений в поведении: видео + аудио, `-shortest`.
- `RenderFrame` — новая функция: на входе только картинка, `-frames:v 1`, на выходе PNG 1920×1080. Аудио не требуется, поэтому кадр рисуется и до выбора трека.

Дальше по слоям:

- `PreviewFrame` в `backend/internal/usecase/youtube.go` — как `Preview`, но без требования `AudioPath` (проверяется только `ImagePath`).
- Эндпоинт `POST /api/youtube/preview-frame`, отдаёт путь к PNG (по аналогии с `ytPreview`).
- Во фронте блок `PlayerPage.tsx:1946-1958` заменяется на `<img>` с этим кадром. Весь CSS-оверлей (`top: 68%`, `4.8cqw`, `blur(22px)`, `scale(1.12)`, `textShadow`) удаляется.

Перерисовка — дебаунс 400мс по изменению обложки, галки оверлея, тайтла, подписи и шрифта. Рендер одного кадра из статичной картинки — порядка 100мс.

Кнопка «Превью видео» (15-секундный клип с аудио) остаётся как есть.

### Часть 2: свои шрифты

В `Settings` (`backend/internal/infrastructure/settings/settings.go`) добавляется одно поле:

```go
YtCustomFonts []string `json:"ytCustomFonts"` // абсолютные пути к .ttf/.otf
```

UI: кнопка «Добавить шрифт» рядом с выпадающим списком открывает диалог выбора файлов (мультивыбор, фильтр `.ttf`/`.otf`), пути дописываются в настройки. Свои шрифты идут в списке после системных, подпись — имя файла без расширения, у каждого крестик для удаления.

Выбранный шрифт кладётся в `YtFont` как путь. `resolveFont` (`ffmpeg.go:236-239`) уже принимает прямой путь к файлу, поэтому ffmpeg-слой не меняется. У своих шрифтов один файл на обе строки (тайтл и подпись) — как сейчас у Impact и Franklin Gothic.

Обработка ошибок: если файл шрифта перемещён или удалён, `resolveFont` молча откатывается на Arial. При рендере кадра существование файла проверяется явно, и текст ошибки показывается там же, где сейчас `previewErr` (`PlayerPage.tsx:1965`) — иначе шрифт тихо не применяется и это выглядит как баг.

## Осознанные компромиссы

- **Пункты списка своих шрифтов отрисованы шрифтом интерфейса.** Сейчас каждый пункт показан своей гарнитурой (`option style={{ fontFamily: f.css }}`). Для своих шрифтов это потребовало бы загрузки файла в вебвью через `@font-face`/`FontFace` + `convertFileSrc`. Не делаем: реальный вид шрифта сразу виден на кадре превью, а он теперь точен.
- **Превью зависит от ffmpeg.** Раньше статичное превью рисовалось без него. Это приемлемо: без ffmpeg вся фича загрузки на YouTube всё равно не работает.
- **Пути к шрифтам, а не копии файлов.** Если пользователь переместит файл, шрифт отвалится — на это есть явная ошибка (см. выше).

## Критерии готовности

1. Статичное превью и кадр итогового видео совпадают пиксель-в-пиксель при одинаковых входных данных.
2. Правка геометрии в `buildChain` меняет превью и видео одновременно, без правок во фронте.
3. Свой `.ttf` можно добавить, выбрать, увидеть на превью и получить в отрендеренном видео.
4. Свои шрифты переживают перезапуск приложения.
5. Удалённый с диска шрифт даёт видимую ошибку, а не тихий откат на Arial.
