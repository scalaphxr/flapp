# Автогенерация SEO-ключевиков в описании YouTube

Дата: 2026-07-18
Статус: утверждён

## Задача

Повторить описание тайп-бит видео с эталона
(`[FREE] *Bankroll Fresh x MexikoDro x Baby Mel Type Beat "Rise" [prod.
BapeBrazy]`) и на максимум задействовать SEO-возможности YouTube. Ключевая
новинка — автогенерация большой «стены» ключевых слов и строки хэштегов прямо
в описании, как в эталоне.

## Разбор эталона

Описание эталона состоит из четырёх частей:

1. **Инфо-блок** — BPM, призыв «leave a like», соцсети, e-mail для WAV/эксклюзива,
   строка лицензии (`[FREE] for non-profit, credit me`), уведомление DMCA. Всё
   статично, меняется только BPM.
2. **Разделитель** — `IGNORE !` и линия из подчёркиваний.
3. **Стена ключевиков** — огромный список через запятую. ~90% — фиксированный
   *ростер* (сеть родственных артистов: ballout, jeezy, chief keef, carti,
   future, plugg… + общие фразы вроде `buy rap beats with hooks for sale` + два
   «вечнозелёных» переиспользуемых блока Drake/Future и Carti). Впереди — только
   артисты конкретного бита.
4. **Хэштеги** — `#BankrollFreshTypeBeat #DrichTypeBeat #JeezyTypeBeat
   #BabyMelTypeBeat #MexikodroTypeBeat` — артисты бита в форме `#ArtistTypeBeat`.

Вывод: стена ключевиков не генерируется заново на каждое видео — это один
большой ростер, который продюсер собрал однажды и переиспользует, вынося вперёд
артистов текущего бита.

## Что уже есть в Flapp

- Шаблоны названия (`{name}/{type}/{bpm}/{key}/{nick}`), дефолт совпадает с
  форматом эталона.
- Шаблон описания с теми же подстановками; в конце — маленькая статичная строка
  ключевиков + хэштеги.
- Поле тегов YouTube (лимит 500 символов) с кнопкой ✨ автогенерации →
  `GenerateTags` (ядро по артистам + живые подсказки YouTube autocomplete,
  бюджет ≤40 тегов / ≤470 символов).
- Обложка: выбор картинки + текстовое наложение ffmpeg.

Движок генерации ключевиков из артистов уже есть — не хватает *ростера* и
подстановок `{keywords}` / `{hashtags}` для описания (где лимит 5000 символов,
на порядок больше, чем в поле тегов).

## Решение

Гибридная модель (утверждена пользователем): артисты бита впереди → личный
ростер → вечнозелёный хвост. Две новые подстановки шаблона и персистентный
ростер.

### Данные (settings.go)

- `YtKeywordRoster string` — пул ключевиков через запятую/перенос строки,
  хранится как введено. Переживает перезапуск.
- `YtRosterAutoGrow bool` — по умолчанию `true`.

Наполнение ростера — «вставить блок + авторост» (утверждено):

- **Вставка**: пользователь один раз вставляет свою готовую стену тегов в
  textarea (вкладка «Общее для всех»). Разбор: split по запятым/переносам,
  trim, отброс пустых, дедуп без учёта регистра.
- **Авторост**: при каждой успешной публикации бита его распознанные
  тип-артисты добавляются в ростер (дедуп). Для нового артиста добавляются
  **обе** формы: голое имя и `{artist} type beat`. Артисты текущей пачки и так
  всегда во фронт-слайсе, поэтому первые видео не бедны на теги.

### Подстановка `{keywords}` (на каждый бит, гибридный порядок)

1. **Фронт-слайс бита** — для каждого тип-артиста из имени файла:
   `{artist} type beat`, `free {artist} type beat`, `{artist} type beat {год}`,
   голое `{artist}`.
2. **Ростер** — весь `YtKeywordRoster` минус то, что уже выдано в шаге 1.
3. **Вечнозелёный хвост** — фиксированный список: `type beat, free type beat,
   instrumental, buy rap beats, rap beats with hooks for sale, trap
   instrumental, rap instrumental, beats`.

Соединение через `, `, дедуп без учёта регистра, обрезка по бюджету ~4000
символов (запас под лимит описания 5000).

### Подстановка `{hashtags}` (на каждый бит)

Тип-артисты бита (без своего ника) → `#` + CamelCase + `TypeBeat`.
`bankroll fresh` → `#BankrollFreshTypeBeat`. Дедуп, **лимит 15** (YouTube
показывает первые 3 над заголовком; >15 или спам-количество скрываются).
Соединение пробелами.

Тип-артисты для обеих подстановок — это распознанные соавторы бита (как в чипсах
диалога), т.е. авторы бита минус свой ник (`bapebrazy type beat` — не запрос,
`bankroll fresh type beat` — запрос).

### Новый дефолтный шаблон описания

Форма как у эталона, e-mail/соцсети пользователь заполняет один раз:

```
{type} Type Beat "{name}"

• BPM: {bpm}  |  Key: {key}
• Prod. {nick} — leave a like if you enjoyed 💯🤝
• Email for WAV / exclusive: your@email.com

[FREE] for non-profit use — you MUST credit (prod. {nick}) in your title.
For profit / exclusive rights: contact me.
Unauthorized use (no lease/exclusive rights) is copyright infringement, subject to DMCA takedown.

IGNORE ↓
________________________________________

{keywords}

{hashtags}
```

### UI (вкладка «Общее для всех»)

Новый блок «Ключевые слова»:

- textarea ростера (вставка/правка),
- переключатель авто-добавления артистов из битов (`YtRosterAutoGrow`),
- маленькое живое превью развёрнутых `{keywords}`/`{hashtags}` для
  сфокусированного бита — чтобы видеть стену до публикации.

Подсказки/справка по шаблонам получают две новые подстановки.

## Границы и архитектура

- Вся логика подстановки — на фронте (`shared/lib/ytKeywords.ts`), рядом с
  существующим `renderYtVars`, чтобы превью и загрузка совпадали дословно и без
  сетевых вызовов. Ростер уже приходит на фронт через settings.
- Бэкенд меняется минимально: два поля в `Settings` + новый `defaultDescription`
  (Go). Существующий `GenerateTags` (живые подсказки) остаётся для поля тегов —
  небольшое дублирование ядра фраз в TS осознанное: стена описания оффлайновая и
  детерминированная.
- Авторост ростера — на фронте в `start()` после успешной публикации бита:
  собрать тип-артистов, слить новые в `YtKeywordRoster`, сохранить через
  `updateSettings`.

## Обработка ошибок / крайние случаи

- Пустой ростер — `{keywords}` всё равно работает (фронт-слайс + хвост).
- Бюджет длины не даёт превысить лимит описания 5000.
- Лимит 15 хэштегов — YouTube прячет описания со спам-количеством хэштегов.
- Дедуп по всему пайплайну без учёта регистра.

## Тестирование

- Юнит-тесты `ytKeywords.ts`: порядок (фронт → ростер → хвост), дедуп, бюджет
  длины, CamelCase-хэштеги, лимит 15, пустой ростер.
- Проверка развёртки `{keywords}`/`{hashtags}` на образце бита + ростере.
- Ручная проверка в диалоге: превью совпадает с тем, что уйдёт в загрузку.
  Реальная публикация на YouTube — не выполняется (необратимо, за пользователем).
